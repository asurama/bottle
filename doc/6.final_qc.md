# 6. 최종 테스트 및 외부 QC 가이드 (Final QC)

## 1. 왜 이 과정이 필요한가? (Rationale)
'Bottle' 프로젝트는 프리미엄 위스키 쉐어링 플랫폼을 지향합니다. 사용자들이 실제 금전(쉐어 비용)을 주고받는 서비스인 만큼, 시스템의 안정성뿐만 아니라 **시각적인 신뢰도(디자인 완성도)**가 매우 중요합니다. GitHub를 통한 외부 QC는 개발환경 이외의 실제 배포와 유사한 환경에서 제3자의 시각으로 문제점을 발견하고 개선하기 위함입니다.

## 2. GitHub 기반 QC 진행 방식
1.  현재 작업 본을 GitHub의 `main` 또는 `release` 브랜치에 Push합니다.
2.  외부 테스터(또는 사용자)가 레포지토리의 소스 및 Preview(배포된 상태)를 확인합니다.
3.  발견된 UI/UX 버그나 개선사항을 Issue 또는 체크리스트에 기록합니다.

## 3. QC 체크리스트 및 검증 항목

### [A] UI/UX 및 디자인 일관성
- [ ] **1. 테마 일관성**: 모든 페이지가 'Amber/Gold + Dark' 테마를 유지하고 있는가?
    - *이유*: 브랜드 아이덴티티 유지 및 프리미엄 느낌 강조.
- [ ] **2. 폰트 및 타이포그래피**: 헤드라인과 본문의 크기, 자간이 가독성 있게 설정되었는가?
    - *이유*: 어두운 배경에서 텍스트 시인성 확보.
- [ ] **3. 이미지 최적화**: 보틀 이미지가 깨지지 않고 적절한 비율(Aspect Ratio)로 로딩되는가?
    - *이유*: 시각적 완성도 및 페이지 로딩 속도 최적화.

### [B] 반응형 디자인 (Responsive)
- [ ] **1. 모바일 뷰어**: 스마트폰(iPhone/Android) 환경에서 카드 UI가 깨지거나 겹치지 않는가?
    - *이유*: 사용자의 절반 이상이 모바일로 접속할 것을 대비.
- [ ] **2. 버튼 터치 영역**: 모바일에서 클릭/터치하기에 충분한 크기인가?
    - *이유*: 모바일 사용 편의성 증대.

### [C] 핵심 비즈니스 로직 및 에러 처리
- [ ] **1. 상태 전이 확인**: '참여 신청 -> 입금 완료 -> 입금 확인' 상태 변경이 DB에 물리적으로 정확히 반영되는가?
    - *이유*: 금전적 오해를 방지하기 위한 가장 핵심적인 신뢰 구간.
- [ ] **2. 예외 상황 처리**: 로그인이 안 된 사용자가 참여 버튼을 눌렀을 때 적절한 안내(Login Redirect)가 나오는가?
    - *이유*: 서비스 이탈 방지 및 사용자 안내 강화.

### [D] 성능 및 보안
- [ ] **1. 환경변수 노출 확인**: `.env`의 민감한 정보(Supabase Key 등)가 클라이언트에 노출되지 않는가?
    - *이유*: 보안 사고 예방.
